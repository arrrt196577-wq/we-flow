# Next Steps: Agent/Subagent Evolution

Date: 2026-06-17

## Recommended Next Step

After the first-stage `AgentSpec + AgentGraphFactory` refactor, the best next move is a small stabilization phase rather than immediately giving `implement_agent` file-writing or command-execution power.

Recommended focus:

1. Verify the real Spring wiring.
2. Define a stable subagent result contract.
3. Make `search_agent` genuinely useful and predictable.
4. Design `implement_agent` permissions before enabling tools.

## 1. Add Real Spring Wiring Verification

The current tests verify the graph behavior and module-level integration, but the next step should add a context-level test that is closer to real application startup.

The test should confirm:

- `lead_agent` can be created through Spring configuration.
- `delegate_task` is registered when delegation is enabled.
- `search_agent` and `implement_agent` are present in `SubAgentRegistry`.
- The lead agent prompt lists `search_agent` and `implement_agent`.
- No circular dependency appears between `LC4jToolService`, `TaskDelegationTool`, `SubAgentRegistry`, and graph-backed subagents.

This is the safest way to lock down the new architecture before adding stronger agent capabilities.

## 2. Define Subagent Output Contract

Before adding more tools, define a stable text contract for subagent results. This helps the lead agent interpret delegated work consistently.

Suggested `search_agent` output:

```text
status: success
summary: ...
evidence:
- source: ...
  finding: ...
recommended_plan:
1. ...
risks:
- ...
```

Suggested failure output:

```text
status: error
code: ...
message: ...
```

The first version can stay as plain text. JSON can come later if the contract needs stricter parsing.

## 3. Strengthen `search_agent`

`search_agent` already has the right first-stage boundary: it can search the web and inspect workspace files, but it cannot write files, execute commands, or delegate tasks.

Next improvements:

- Make its system prompt more explicit about when to use workspace search versus web search.
- Require evidence paths or source links for factual claims.
- Require a concrete recommended plan when the task is implementation-related.
- Tell it not to modify files, run commands, or call `delegate_task`.
- Add tests that prove it only sees `web_search`, `find_files`, `read_file`, and `list_dir`.

This makes `search_agent` useful as a factual planning agent before `implement_agent` becomes active.

## 4. Design `implement_agent` Before Enabling Tools

Do not give `implement_agent` arbitrary shell access as the next step.

Design the permission boundary first:

- Read-only tools: `find_files`, `read_file`, `list_dir`.
- Write tool: prefer a patch-oriented tool such as `apply_patch` rather than broad file writes.
- Command tool: use an allowlist first.

Initial command allowlist can include:

- `rg`
- `mvn test`
- `mvn -pl ... test`
- `mvn -pl ... -am test`
- `git diff`
- `git status --short`

Avoid enabling destructive commands, recursive deletes, arbitrary move operations, or unrestricted shell execution in the first implementation pass.

## Suggested Sequence

1. Add Spring wiring/context test for the new lead/subagent setup.
2. Improve `search_agent` prompt and result contract.
3. Add tests for `search_agent` output and tool visibility.
4. Design `implement_agent` tool policy and command allowlist.
5. Only then enable write/command tools for `implement_agent`.

This sequence keeps the architecture moving toward a DeerFlow-style model while avoiding a risky jump into unrestricted implementation tooling.
